# 履修登録システム リバースエンジニアリング報告書

**対象:** 京都芸術大学附属高等学校 通信制課程 履修登録システム  
**分析日:** 2026-04-08  
**目的:** 自校への応用のためのシステム理解・技術的負債の特定

---

## 1. システム概要

### 1.1 技術スタック
- **バックエンド:** Google Apps Script (V8ランタイム)
- **フロントエンド:** 単一HTML（CSS/JS埋込）、GAS WebApp として配信
- **データストア:** Google Spreadsheet（7シート）
- **認証:** Google Workspace ドメイン認証 (`Session.getActiveUser().getEmail()`)
- **キャッシュ:** 2段階（インメモリ + GAS ScriptCache、TTL 5分）

### 1.2 規模
| 項目 | 数値 |
|------|------|
| バックエンド (コード.js) | 5,046行、134関数 |
| フロントエンド (index.html) | 19,224行（CSS 1,623行 + HTML 1,462行 + JS 16,119行） |
| スプレッドシート | 7シート、最大78列 |
| 科目数 | 約68科目 |
| モーダル数 | 28個 |
| 管理オブジェクト | 14個（Manager/Screen） |

### 1.3 ユーザーロール
| ロール | 権限 | 判定方法 |
|--------|------|----------|
| admin（管理者） | 全権限 | 教職員データ.権限 = "管理者" |
| kyomu（教務） | 教務部権限 | 教職員データ.権限 = "教務" |
| teacher（教員） | 担任権限 | 教職員データに存在、権限空欄 |
| student（生徒） | 自身の登録のみ | 上記以外（生徒用ドメインで判定） |

### 1.4 主要機能
- **生徒向け:** 履修科目の選択・仮登録・本登録
- **教員向け:** 担当生徒の登録状況確認・承認・差戻し
- **管理者向け:** 科目管理、教職員管理、学生管理、設定管理、年度移行、一括印刷

## 2. データフロー

```
[生徒/教員ブラウザ]
    ↓ GAS WebApp (doGet)
[index.html] ←→ [google.script.run] ←→ [コード.js]
                                            ↓
                                    [Google Spreadsheet]
                                    ├─ 科目データ（マスタ）
                                    ├─ 登録データ（トランザクション）
                                    ├─ 教職員データ（ユーザー管理）
                                    ├─ 教科書（マスタ）
                                    ├─ レポート（マスタ）
                                    ├─ SM設定（スクーリンググループ）
                                    └─ 設定（Key-Value）
```

### 認証フロー
1. `doGet()` でHTMLを返却（tokenをURLパラメータから取得）
2. フロントエンドで `getInitialDataWithAuth()` を呼び出し
3. `Session.getActiveUser().getEmail()` でメール取得
4. 教職員データシートとメール照合 → ロール決定
5. ロールに応じた画面表示

### 登録フロー
1. 生徒が科目を選択 → `saveRegistrationData()` で一時保存
2. `submitRegistrationData()` で仮登録（ステータス = "仮登録"）
3. 教員が確認 → `approveRegistration()` で承認
4. 管理者が `setFinalApproval()` で本登録許可
5. 生徒が `finalRegister()` で本登録（認証コード必要）

## 3. スプレッドシート構造

（詳細は `docs/spreadsheet-structure.md` を参照）

### シート間の関係
```
科目データ.科目名 ──┬── 登録データ.各科目カラム名（横持ち）
                    ├── 教科書.科目名
                    ├── レポート.科目名
                    └── SM設定.科目名

教職員データ.メールアドレス ── 認証（Session.getActiveUser()）
登録データ.メールアドレス ── 認証（Session.getActiveUser()）
登録データ.学籍番号 ── 科目データ.表示/非表示/抽選学籍番号
```

**重要:** 結合キーは全て「科目名」（日本語文字列）。正規化されたIDは使用されていない。

## 4. 技術的負債一覧

### 4.1 重大度: Critical

#### [BUG] approveStudent関数が未定義
- **場所:** コード.js:2616
- **内容:** `approveRegistration()` が `approveStudent(studentId)` を呼び出すが、この関数は定義されていない
- **影響:** 教員による承認機能が動作しない可能性

#### [ARCH] 全コードが2ファイルに集中
- **バックエンド:** 5,046行・134関数が `コード.js` 1ファイル
- **フロントエンド:** 19,224行（CSS+HTML+JS）が `index.html` 1ファイル（1.1MB）
- **影響:** 保守性・可読性が極端に低い、複数人での開発が困難

### 4.2 重大度: High

#### [DRY] キャッシュ取得パターンの4重複
- **場所:** コード.js:2122, 2176, 2222, 2284
- **内容:** `getXxxDataCached()` が4つとも同一パターン（メモリ→ScriptCache→Sheet読込）

#### [DRY] ヘッダーインデックス取得パターンの20+重複
- **場所:** コード.js 全体に散在
- **内容:** `headers.indexOf('学籍番号')` 等が20箇所以上で繰り返し

#### [DRY] 行検索パターンの10+重複
- **場所:** コード.js:1338, 1490, 1814 他
- **内容:** `for (let i = 1; ...) if (data[i][idx] === value)` パターン

#### [DRY] 不可視文字削除の5重複定義（フロントエンド）
- **場所:** index.html:11027, 11275, 11528, 12270, 12624
- **内容:** `stripInvisible()` がローカル関数として5回定義

#### [SEC] innerHTML使用（60+箇所、XSSリスク）
- **場所:** index.html 全体に散在
- **内容:** `Utils.escapeHtml()` は存在するが多くの箇所で未使用
- **特に危険:** ユーザー入力（名前等）を含むHTML生成（index.html:5544, 6790, 7249）

#### [PERF] O(n^2)検索パターン
- **場所:** コード.js:1700付近 (getAllStudentsDataInternal)
- **内容:** submissionData × rosterData のネストループ
- **改善:** Map化で O(n) に

### 4.3 重大度: Medium

#### [HARDCODE] マジックナンバー・固定文字列
- コード.js: 基本カラム名リスト（1118行）、初期設定値（164-170行）
- index.html: 30単位上限（1738行）、カラーコード50+箇所、ブレークポイント（19040-19042行）

#### [STYLE] インラインスタイル780+箇所（フロントエンド）
- CSS変数（`--primary-color`等）が定義されているが、インラインstyleで直接カラーコードを指定

#### [ARCH] モーダル28個がコンポーネント化されていない
- ほぼ同一構造のモーダルが個別に定義

#### [ARCH] イベントハンドラが分散
- `onclick`属性（20+箇所）と `addEventListener`（各Manager内）が混在

#### [ARCH] グローバル変数汚染
- `AppState`（35+プロパティ）、`cachedAllStudentsData`、`cachedTeacherEmails` 等

#### [SEC] 認証コードのブルートフォース対策なし
- コード.js:2926 — 単純な一致チェックのみ

#### [SEC] 権限がスプレッドシートのデータに依存
- 教職員シートの改ざんで権限昇格の可能性

#### [DATA] 登録データが横持ちテーブル
- 科目追加のたびにカラム追加が必要（78列）
- 正規化されていない

#### [DATA] 科目名が結合キー
- 日本語文字列が全シートのリレーションキー
- 科目名変更時の影響範囲が大きい

### 4.4 重大度: Low

#### [ERR] エラーハンドリングの不統一
- Throwパターン、Booleanパターン、Try-catchパターンが混在

#### [PERF] JSON.parse/stringify の多用
- キャッシュの保存・復元で毎回実行（約10箇所）

#### [STYLE] ローディングスピナーが複数箇所で異なるスタイル定義

### 5.3 共通化すべきパターン（優先順）

| 優先度 | パターン | 重複回数 | 削減見込み |
|--------|---------|----------|-----------|
| P1 | キャッシュラッパー関数 | 4回 | ~120行 |
| P1 | ヘッダーMap構築 + 行検索 | 30+回 | ~200行 |
| P1 | stripInvisible() のグローバル化 | 5回 | ~25行 |
| P2 | モーダルコンポーネント | 28個 | ~500行 |
| P2 | インラインスタイル → CSSクラス | 780箇所 | 可読性大幅向上 |
| P2 | シート遅延取得の共通化 | 5回 | ~50行 |
| P3 | innerHTML → textContent + escapeHtml | 60箇所 | セキュリティ向上 |

### 5.4 自校応用時の設計改善提案

1. **科目IDの導入** — 科目名ではなく一意のIDをキーにする
2. **登録データの縦持ち化** — 科目をカラムではなく行として持つ（拡張性向上）
3. **設定のスキーマ定義** — Key-Valueではなく型付き設定オブジェクト
4. **認証コードの強化** — レート制限、有効期限の追加
5. **テストの追加** — GAS用テストフレームワーク（clasp + jest）の導入

## 6. 実装の優れている点（参考にすべき点）

1. **2段階キャッシュ戦略** — メモリ + ScriptCache でGASの実行時間制限に対応
2. **遅延ローディング** — シート参照の遅延取得でパフォーマンス最適化
3. **初期データ一括取得** — `getInitialDataWithAuth()` で複数API呼び出しを1回に削減
4. **権限ガード関数** — `requireAdmin()`, `requireTeacherOrAdmin()` でアクセス制御
5. **リトライ機構** — `getDataWithRetry()` でスプレッドシートAPI の一時的エラーに対応
6. **メンテナンスモード** — 設定シートで動的にアクセス制限可能
7. **クラス情報の表示制御** — 設定で生徒への表示形式を切替可能
8. **年度移行機能** — 年度データの引継ぎ処理が実装済み
